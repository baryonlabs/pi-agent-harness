# Harness — pi Agent Team & Skill Architect

도메인/프로젝트에 맞는 pi 하네스를 구성하고, 각 서브에이전트의 역할을 정의하며, 에이전트가 사용할 스킬과 오케스트레이션 프롬프트를 생성하는 메타 스킬.

**핵심 원칙:**
1. 에이전트 정의(`.pi/agents/`), 스킬(`.pi/skills/`), 오케스트레이션 프롬프트(`.pi/prompts/`)를 생성한다.
2. **subagent 도구 오케스트레이션을 기본 실행 모드로 사용한다.** — pi에는 빌트인 서브에이전트·실시간 팀 메시징이 없으므로, 번들된 `subagent` 확장의 single/parallel/chain 모드와 `_workspace/` 파일 핸드오프로 협업을 구현한다.
3. **AGENTS.md에 하네스 포인터를 등록한다.** — 새 세션에서 오케스트레이션 프롬프트가 트리거되도록 최소한의 포인터(트리거 규칙 + 변경 이력)만 기록한다.
4. **하네스는 고정물이 아니라 진화하는 시스템이다.** — 매 실행 후 피드백을 반영하고, 에이전트·스킬·프롬프트·AGENTS.md를 지속 갱신한다.
5. **프로세스를 작업 크기에 맞추고, 사람이 통제권을 쥔다.** — 요청을 복잡도로 트리아지하여 필요한 만큼만 위임하고(간단한 요청에 팀을 띄우지 않는다), 커밋·PR·머지·배포 등 되돌리기 어려운 행동은 명시적 요청 없이 하지 않는다. 상세 패턴(트리아지·구조화된 핸드오프·검증 게이트·바운디드 리페어·HITL)은 `references/orchestrator-template.md`의 "채용 패턴" + `references/agent-design-patterns.md`의 "구조화된 핸드오프 스키마" 참조.

> **pi ≠ Claude Code.** pi는 의도적으로 미니멀하다: 빌트인 서브에이전트·MCP·플랜모드·투두·백그라운드 bash가 없다. 모든 확장은 skills/prompt templates/extensions/packages로 구현한다. 따라서 "에이전트 팀"은 실시간 메시징 팀이 아니라 **별도 pi 프로세스를 스폰하는 subagent 오케스트레이션**으로 매핑된다. 자세한 매핑은 `references/agent-design-patterns.md` 참조.

## 전제: subagent 확장

이 하네스 패키지는 `extensions/subagent/`에 subagent 확장을 번들한다. 이 확장이 `subagent` 도구를 등록하여 single/parallel/chain 위임을 가능하게 한다. 하네스 패키지가 설치되면(`pi install ...`) 확장이 자동 로드되므로 생성된 하네스는 추가 설치 없이 동작한다.

- **agentScope:** `subagent` 도구는 기본적으로 사용자 레벨 에이전트(`~/.pi/agent/agents/`)만 로드한다. 프로젝트 레벨 `.pi/agents/*.md`를 사용하려면 도구 호출 시 `agentScope: "both"`(또는 `"project"`)를 넘겨야 한다. **오케스트레이션 프롬프트에 반드시 `agentScope: "both"`를 명시한다.**
- **모델:** pi는 에이전트별로 `model` frontmatter를 지정한다. 표준 ID: `claude-opus-4-7`(고난도 추론), `claude-sonnet-4-5`(일반 구현/리뷰), `claude-haiku-4-5`(경량 정찰/추출). 추론 품질이 결과를 좌우하는 에이전트는 opus, 단순 수집은 haiku로 티어링한다.

## 워크플로우

### Phase 0: 현황 감사

하네스 스킬이 트리거되면 가장 먼저 기존 하네스 현황을 확인한다.

1. `프로젝트/.pi/agents/`, `프로젝트/.pi/skills/`, `프로젝트/.pi/prompts/`, `프로젝트/AGENTS.md`(또는 `CLAUDE.md`)를 읽는다
2. 현황에 따라 실행 모드를 분기한다:
   - **신규 구축**: 에이전트/스킬 디렉토리가 없거나 비어있음 → Phase 1부터 전체 실행
   - **기존 확장**: 기존 하네스가 있고 새 에이전트/스킬 추가 요청 → 아래 Phase 선택 매트릭스에 따라 필요한 Phase만 실행
   - **운영/유지보수**: 기존 하네스의 감사·수정·동기화 요청 → Phase 7-5 운영/유지보수 워크플로우로 이동

   **기존 확장 시 Phase 선택 매트릭스:**
   | 변경 유형 | Phase 1 | Phase 2 | Phase 3 | Phase 4 | Phase 5 | Phase 6 |
   |----------|---------|---------|---------|---------|---------|---------|
   | 에이전트 추가 | 건너뜀 (Phase 0 결과 활용) | 배치 결정만 | 필수 (3-0 포함) | 전용 스킬 필요 시 (4-0 포함) | 오케스트레이션 프롬프트 수정 | 필수 |
   | 스킬 추가/수정 | 건너뜀 | 건너뜀 | 건너뜀 | 필수 (4-0 포함) | 연결 변경 시 | 필수 |
   | 아키텍처 변경 | 건너뜀 | 필수 | 영향받는 에이전트만 (3-0 포함) | 영향받는 스킬만 (4-0 포함) | 필수 | 필수 |
3. 기존 에이전트/스킬/프롬프트 목록과 AGENTS.md 기록을 대조하여 불일치(drift)를 감지한다
4. 감사 결과를 사용자에게 요약 보고하고, 실행 계획을 확인받는다

### Phase 1: 도메인 분석
1. 사용자 요청에서 도메인/프로젝트 파악
2. 핵심 작업 유형 식별 (생성, 검증, 편집, 분석 등)
3. Phase 0 감사 결과를 기반으로 기존 에이전트/스킬과의 충돌/중복 분석
4. 프로젝트 코드베이스 탐색 — 기술 스택, 데이터 모델, 주요 모듈 파악
5. **사용자 숙련도 감지** — 대화의 맥락 단서(사용 용어, 질문 수준)로 기술 수준을 파악하고, 이후 커뮤니케이션 톤을 조절한다. 코딩 경험이 적은 사용자에게는 "assertion", "JSON schema" 같은 용어를 설명 없이 쓰지 않는다.

### Phase 2: 팀 아키텍처 설계

#### 2-1. 위임 모드 선택

**subagent 오케스트레이션이 기본값이다.** pi에는 실시간 팀 메시징이 없으므로, 협업은 메인 에이전트가 `subagent` 도구로 위임하고 `_workspace/` 파일로 결과를 주고받는 형태로 구현한다. 위임 모드는 세 가지다:

| 모드 | 언제 사용 | 도구 파라미터 |
|------|----------|--------------|
| **single** (단일) | 한 에이전트가 격리 컨텍스트에서 단발 작업 | `{ agent, task }` |
| **parallel** (병렬) | 독립적 작업을 동시 수행 (팬아웃/팬인) | `{ tasks: [{agent, task}, ...] }` (최대 8개, 동시 4개) |
| **chain** (연쇄) | 순차 의존 작업, 이전 출력을 다음 입력으로 (파이프라인) | `{ chain: [{agent, task}, ...] }`, `{previous}` 플레이스홀더 |

**의사결정 순서:**
1. 작업이 순차 의존이면 → **chain**
2. 독립 병렬이면 → **parallel**
3. 단일 격리 작업이면 → **single**
4. 복합이면 → 오케스트레이션 프롬프트에서 모드를 조합 (예: parallel 수집 → 메인이 통합 → chain 검증)

> pi에는 실시간 에이전트 간 통신(SendMessage)이 없다. "팀원 간 토론"이 필요했던 패턴은 **메인 에이전트가 중간 산출물을 읽고 다음 위임을 조정**하는 방식으로 대체한다. 상세 비교와 패턴별 의사결정 트리는 `references/agent-design-patterns.md` 참조.

#### 2-2. 아키텍처 패턴 선택

1. 작업을 전문 영역으로 분해
2. 팀 구조 결정 (아키텍처 패턴은 `references/agent-design-patterns.md` 참조)
   - **파이프라인** → chain 모드: 순차 의존 작업
   - **팬아웃/팬인** → parallel 모드: 병렬 독립 작업
   - **전문가 풀** → single 모드: 상황별 선택 호출
   - **생성-검증** → chain 모드: worker → reviewer → worker
   - **감독자** → 메인 에이전트가 진행 상황을 보며 subagent 동적 호출
   - **계층적 위임** → 메인이 상위 작업을, 위임된 에이전트가 하위 작업을 (깊이 2단계 권장)

#### 2-3. 에이전트 분리 기준

전문성·병렬성·컨텍스트·재사용성 4축으로 판단한다. 상세 기준표는 `references/agent-design-patterns.md`의 "에이전트 분리 기준" 참조. 기존 에이전트와의 중복·재사용 검토는 Phase 3-0에서 다룬다.

### Phase 3: 에이전트 정의 생성

#### 3-0. 기존 에이전트 중복 검토

신규 에이전트 생성 전, `프로젝트/.pi/agents/`의 기존 에이전트와 중복 여부를 확인한다. 하네스를 반복 구축하다 보면 역할이 겹치는 에이전트가 다른 이름으로 누적되기 쉽다.

> 중복 분류 기준과 재사용 설계는 `references/agent-design-patterns.md`의 "에이전트 재사용 설계" 참조.

**모든 에이전트는 반드시 `프로젝트/.pi/agents/{name}.md` 파일로 정의한다.** subagent 도구의 task에 역할을 직접 길게 넣는 것은 금지한다. 이유:
- 에이전트 정의가 파일로 존재해야 다음 세션에서 재사용 가능
- `subagent` 확장이 `.pi/agents/*.md`를 자동 발견하여 `agent` 이름으로 호출 가능
- 하네스의 핵심 가치는 에이전트(누가)와 스킬(어떻게)의 분리

**에이전트 정의 형식 (pi `subagent` 확장 규약):**

```markdown
---
name: agent-name
description: 이 에이전트가 무엇을 하는지 (subagent 도구가 선택 시 참조)
tools: read, grep, find, ls, bash
model: claude-sonnet-4-5
---

여기 본문이 곧 에이전트의 시스템 프롬프트가 된다. 역할·작업 원칙·입출력 프로토콜을 담는다.
```

- **tools:** 공백/쉼표 구분 화이트리스트. 생략하면 기본 도구(read/write/edit/bash) 전체. 읽기 전용 분석가는 `read, grep, find, ls`로 제한하여 실수로 파일을 수정하지 못하게 한다 (pi엔 `Explore` 빌트인 타입이 없으므로 도구 화이트리스트로 동일 효과를 낸다).
- **model:** 추론 난도에 맞춰 티어링한다. 정찰/추출=`claude-haiku-4-5`, 일반 구현/리뷰=`claude-sonnet-4-5`, 고난도 설계/통합=`claude-opus-4-7`.

> 정의 템플릿과 실제 파일 전문은 `references/agent-design-patterns.md`의 "에이전트 정의 구조" + `references/team-examples.md` 참조.

**QA 에이전트 포함 시 필수 사항:**
- QA 에이전트는 `tools`에 `read, grep, find, ls, bash`를 부여한다 (패턴 검색·교차 대조에 필요). 읽기 전용으로 두되 bash는 read-only 명령만 쓰도록 본문에 명시
- QA의 핵심은 "존재 확인"이 아니라 **"경계면 교차 비교"** — API 응답과 프론트 훅을 동시에 읽고 shape을 비교
- QA는 전체 완성 후 1회가 아니라, **각 모듈 완성 직후 점진적으로 실행** (incremental QA)
- 상세 가이드: `references/qa-agent-guide.md` 참조

### Phase 4: 스킬 생성

각 에이전트가 사용할 스킬을 `프로젝트/.pi/skills/{name}/SKILL.md`에 생성한다. pi는 [Agent Skills 표준](https://agentskills.io/specification)을 따르므로 SKILL.md 형식은 Claude Code와 거의 동일하다. 상세 작성 가이드는 `references/skill-writing-guide.md` 참조.

#### 4-0. 기존 스킬 중복 검토

신규 스킬 생성 전, `프로젝트/.pi/skills/`의 기존 스킬과 중복 여부를 확인한다.

> 중복 분류 기준과 일반화 패턴은 `references/skill-writing-guide.md`의 "스킬 재사용 설계" 참조.

#### 4-1. 스킬 구조

```
skill-name/
├── SKILL.md (필수)
│   ├── YAML frontmatter (name, description 필수)
│   └── Markdown 본문
└── Bundled Resources (선택)
    ├── scripts/    - 반복/결정적 작업용 실행 코드
    ├── references/ - 조건부 로딩하는 참조 문서
    └── assets/     - 출력에 사용되는 파일 (템플릿, 이미지 등)
```

#### 4-2. Description 작성 — 적극적 트리거 유도

description은 스킬이 시스템 프롬프트에 노출되는 유일한 트리거 단서다. 모델은 트리거를 보수적으로 판단하므로 description을 **적극적("pushy")**으로 작성한다. (강제 로딩이 필요하면 `/skill:name` 커맨드로도 호출 가능.)

**나쁜 예:** `"PDF 문서를 처리하는 스킬"`
**좋은 예:** `"PDF 파일 읽기, 텍스트/테이블 추출, 병합, 분할, 회전, 워터마크, 암호화, OCR 등 모든 PDF 작업을 수행. .pdf 파일을 언급하거나 PDF 산출물을 요청하면 반드시 이 스킬을 사용할 것."`

핵심: 스킬이 하는 일 + 구체적 트리거 상황을 모두 기술하고, 유사하지만 트리거하면 안 되는 경우와 구분되도록 작성.

#### 4-3. 본문 작성 원칙

| 원칙 | 설명 |
|------|------|
| **Why를 설명하라** | "ALWAYS/NEVER" 같은 강압적 지시 대신, 왜 그렇게 해야 하는지 이유를 전달한다. 모델은 이유를 이해하면 엣지 케이스에서도 올바르게 판단한다. |
| **Lean하게 유지** | 컨텍스트 윈도우는 공공재다. SKILL.md 본문은 500줄 이내를 목표로, 무게를 벌지 않는 내용은 삭제하거나 references/로 이동한다. |
| **일반화하라** | 특정 예시에만 맞는 좁은 규칙보다, 원리를 설명하여 다양한 입력에 대응할 수 있게 한다. 오버피팅 금지. |
| **반복 코드는 번들링** | 에이전트들이 공통으로 작성하는 스크립트가 발견되면 `scripts/`에 미리 번들링한다. 스킬은 상대 경로로 스크립트를 참조한다. |
| **명령형으로 작성** | "~한다", "~하라" 형태의 명령형/지시형 어조를 사용한다. |

#### 4-4. Progressive Disclosure (단계적 정보 공개)

스킬은 3단계 로딩 시스템으로 컨텍스트를 관리한다:

| 단계 | 로딩 시점 | 크기 목표 |
|------|----------|----------|
| **Metadata** (name + description) | 항상 시스템 프롬프트에 존재 | ~100단어 |
| **SKILL.md 본문** | 스킬 트리거 또는 `/skill:name` 시 `read`로 로드 | <500줄 |
| **references/** | 필요할 때만 | 무제한 (스크립트는 로딩 없이 실행 가능) |

**크기 관리 규칙:**
- SKILL.md가 500줄에 근접하면 세부 내용을 references/로 분리하고, 본문에 "언제 이 파일을 읽으라"는 포인터를 남긴다
- 300줄 이상의 reference 파일에는 상단에 **목차(ToC)**를 포함한다
- 도메인/프레임워크별 변형이 있으면 references/ 하위에 도메인별로 분리하여, 관련 파일만 로드한다

#### 4-5. 스킬-에이전트 연결 원칙

- 에이전트 1개 ↔ 스킬 1~N개 (1:1 또는 1:다)
- 여러 에이전트가 공유하는 스킬도 가능
- 스킬은 "어떻게 하는가"를 담고, 에이전트는 "누가 하는가"를 담는다
- 위임된 서브에이전트는 격리된 pi 프로세스이므로, 사용할 스킬을 **에이전트 본문(시스템 프롬프트)에 명시**하거나 task에 `/skill:name`을 포함해 로드를 유도한다

> 상세 작성 패턴, 예시, 데이터 스키마 표준은 `references/skill-writing-guide.md` 참조.

### Phase 5: 통합 및 오케스트레이션

오케스트레이션 프롬프트는 개별 에이전트와 스킬을 하나의 워크플로우로 엮는다. Phase 4의 스킬이 "각 에이전트가 무엇을 어떻게 하는가"라면, 오케스트레이션 프롬프트는 "누가 언제 어떤 순서로 협업하는가"를 정의한다.

pi에서 오케스트레이터는 **프롬프트 템플릿**(`프로젝트/.pi/prompts/{name}.md`)으로 구현한다 — 사용자가 `/{name}`으로 호출하면 메인 에이전트가 `subagent` 도구를 single/parallel/chain으로 호출하도록 지시하는 마크다운이다. 워크플로우가 복잡해 단계적 정보 공개가 필요하면, 얇은 프롬프트 + 상세 절차를 담은 스킬(`.pi/skills/{domain}-orchestrator/`)을 함께 둔다. 구체적 템플릿은 `references/orchestrator-template.md` 참조.

**기존 확장 시 수정:** 신규 구축이 아닌 기존 확장일 때는 오케스트레이션 프롬프트를 새로 만들지 않고 기존 것을 수정한다. 에이전트 추가 시 위임 흐름·작업 할당에 새 에이전트를 반영하고, description에 새 트리거 키워드를 추가한다.

#### 5-0. 오케스트레이션 프롬프트 패턴 (모드별)

오케스트레이션 프롬프트는 메인 에이전트에게 `subagent` 도구 사용법을 지시한다. **항상 `agentScope: "both"`를 명시**하여 프로젝트 `.pi/agents/`가 발견되게 한다.

```markdown
---
description: {도메인} 워크플로우 — {요약}
argument-hint: "<입력>"
---
subagent 도구를 사용해 다음 워크플로우를 실행하라 (agentScope: "both"):

1. parallel로 {agent-a}와 {agent-b}를 동시에 호출하여 $@ 에 대한 자료를 수집한다.
2. 두 결과를 _workspace/ 에 저장하고 메인이 읽어 통합 초안을 만든다.
3. chain으로 {worker} → {reviewer} → {worker}를 실행하여 초안을 다듬는다 ({previous} 전달).

최종 산출물은 {output-path}에 저장한다.
```

- **파이프라인** → `chain` 한 번. `{previous}`로 단계 간 출력 전달.
- **팬아웃/팬인** → `parallel` 수집 후, 메인이 결과를 읽어 통합 (pi엔 통합 전용 "팀 합의"가 없으므로 메인이 통합자 역할).
- **생성-검증** → `chain`: worker → reviewer → worker (pi 공식 `implement-and-review` 패턴).
- **감독자** → 메인이 `subagent` 도구를 루프 안에서 동적 호출하며 진행 상황에 따라 다음 task 결정.

#### 5-1. 데이터 전달 프로토콜

| 전략 | 방식 | 적합한 경우 |
|------|------|-----------|
| **반환값 기반** | `subagent` 도구의 최종 출력(텍스트)을 메인이 수집 | 결과를 메인이 직접 받아 다음 단계 결정 |
| **chain `{previous}`** | 이전 단계 출력이 다음 task에 자동 주입 | 순차 파이프라인 |
| **파일 기반** | 약속된 `_workspace/` 경로에 쓰고 읽음 | 대용량 데이터, 구조화된 산출물, 감사 추적 (parallel 출력은 task당 50KB로 캡되므로 큰 산출물은 파일로) |

**권장 조합:** chain `{previous}`(단계 전달) + 파일 기반(대용량·감사) + 반환값(메인의 통합 판단)

파일 기반 전달 시 규칙:
- 작업 디렉토리 하위에 `_workspace/` 폴더를 만들어 중간 산출물 저장
- 파일명 컨벤션: `{phase}_{agent}_{artifact}.{ext}` (예: `01_analyst_requirements.md`)
- 최종 산출물만 사용자 지정 경로에 출력, 중간 파일(`_workspace/`)은 보존 (사후 검증·감사 추적용)

#### 5-2. 에러 핸들링

오케스트레이션 프롬프트에 에러 처리 방침을 포함한다. 핵심 원칙: 1회 재시도 후 재실패 시 해당 결과 없이 진행(보고서에 누락 명시), 상충 데이터는 삭제하지 않고 출처 병기. chain 모드는 첫 실패 단계에서 멈추므로, 메인이 부분 결과로 복구할지 판단한다.

> 에러 유형별 전략표와 구현 상세는 `references/orchestrator-template.md`의 "에러 핸들링" 참조.

#### 5-3. 팀 크기 가이드라인

| 작업 규모 | 권장 에이전트 수 | 에이전트당 작업 수 |
|----------|------------|--------------|
| 소규모 (5~10개 작업) | 2~3명 | 3~5개 |
| 중규모 (10~20개 작업) | 3~5명 | 4~6개 |
| 대규모 (20개+ 작업) | 5~7명 | 4~5개 |

> parallel은 최대 8개 task, 동시 4개로 제한된다. 큰 팬아웃은 배치로 나눠 호출한다. 에이전트가 많을수록 메인의 통합 부담이 커진다 — 3명의 집중된 에이전트가 5명의 산만한 에이전트보다 낫다.

#### 5-4. AGENTS.md 하네스 포인터 등록

하네스 구성 완료 후, 프로젝트의 `AGENTS.md`(pi의 기본 컨텍스트 파일, `CLAUDE.md`도 함께 로드됨)에 최소한의 포인터를 등록한다. AGENTS.md는 새 세션마다 로딩되므로, 하네스 존재와 트리거 규칙만 기록하면 오케스트레이션 프롬프트가 나머지를 처리한다.

**AGENTS.md 템플릿:**

````markdown
## 하네스: {도메인명}

**목표:** {하네스의 핵심 목표 한 줄}

**트리거:** {도메인} 관련 작업은 `/{orchestrator-prompt-name}` 프롬프트로 실행한다. 단순 질문은 직접 응답 가능.

**변경 이력:**
| 날짜 | 변경 내용 | 대상 | 사유 |
|------|----------|------|------|
| {YYYY-MM-DD} | 초기 구성 | 전체 | - |
````

**AGENTS.md에 넣지 않는 것:** 에이전트 목록, 스킬 목록, 디렉토리 구조, 실행 규칙 상세. 이유: 에이전트/스킬 목록은 `.pi/agents/`, `.pi/skills/`에서 관리하므로 중복이다. AGENTS.md는 **포인터(트리거 규칙) + 변경 이력**만 담는다.

#### 5-5. 후속 작업 지원

오케스트레이션 프롬프트는 초기 실행뿐 아니라 후속 작업도 처리해야 한다:

**1. description에 후속 키워드 포함:** "다시 실행", "재실행", "업데이트", "수정", "보완", "{도메인}의 {부분작업}만 다시", "이전 결과 기반으로", "결과 개선".

**2. 워크플로우 시작에 컨텍스트 확인 단계:** `_workspace/` 존재 여부로 초기/새/부분 재실행을 판별 (Phase 5-4 오케스트레이터 템플릿의 "Phase 0: 컨텍스트 확인" 참조).

**3. 에이전트 정의에 재호출 지침:** 각 에이전트 본문에 "이전 산출물이 있으면 읽고 개선점을 반영한다"를 명시.

> 오케스트레이터 템플릿의 "Phase 0: 컨텍스트 확인" 섹션 참조: `references/orchestrator-template.md`

### Phase 6: 검증 및 테스트

생성된 하네스를 검증한다. 상세 테스트 방법론은 `references/skill-testing-guide.md` 참조.

#### 6-1. 구조 검증

- 모든 에이전트 파일이 `.pi/agents/`에 있고 frontmatter(`name`, `description`, 선택 `tools`/`model`)가 유효한지 확인
- 스킬의 frontmatter(`name`, `description`) 검증 — pi는 description 누락 스킬을 로드하지 않음
- 오케스트레이션 프롬프트가 `.pi/prompts/`에 있고 `subagent` 도구 + `agentScope: "both"`를 명시하는지 확인
- 에이전트 이름과 오케스트레이션 프롬프트가 참조하는 이름이 일치하는지 확인

#### 6-2. 위임 모드별 검증

- **single**: agent 이름이 실제 정의와 일치, task가 자족적인지
- **parallel**: task 수 ≤ 8, 각 task가 독립적인지, 출력 50KB 초과 시 파일 저장 지시가 있는지
- **chain**: `{previous}` 플레이스홀더가 단계마다 올바르게 연결되는지, 첫 단계가 입력을 받는지

#### 6-3. 스킬 실행 테스트

생성된 각 스킬에 대해 실제 실행 테스트를 수행한다:

1. **테스트 프롬프트 작성** — 각 스킬에 대해 2~3개의 현실적인 테스트 프롬프트를 작성한다.
2. **With-skill vs Without-skill 비교** — `subagent` 도구로 두 위임을 실행: 하나는 스킬을 로드(`/skill:name` 또는 본문 지시), 하나는 스킬 없이. 산출물 품질 차이로 스킬의 부가가치를 확인.
3. **결과 평가** — 정성적(사용자 리뷰) + 정량적(assertion). 객관적 검증이 가능하면(파일 생성, 데이터 추출) assertion을 정의.
4. **반복 개선 루프** — 문제 발견 시 피드백을 **일반화**하여 스킬을 수정하고 재테스트. 만족하거나 의미 있는 개선이 없을 때까지 반복.
5. **반복 패턴 번들링** — 에이전트들이 공통으로 작성하는 코드가 발견되면 `scripts/`에 미리 번들링.

#### 6-4. 트리거 검증

각 스킬과 오케스트레이션 프롬프트의 description이 올바르게 트리거되는지 검증한다:

1. **Should-trigger 쿼리** (8~10개) — 트리거해야 하는 다양한 표현 (공식적/캐주얼, 명시적/암시적)
2. **Should-NOT-trigger 쿼리** (8~10개) — 키워드가 유사하지만 다른 도구/스킬이 적합한 "near-miss" 쿼리

**near-miss 작성 핵심:** 명백히 무관한 쿼리는 테스트 가치가 없다. **경계가 모호한 쿼리**가 좋은 테스트 케이스다. 기존 스킬과의 트리거 충돌도 이 단계에서 확인한다.

#### 6-5. 드라이런 테스트

- 오케스트레이션 프롬프트의 단계 순서가 논리적인지 검토
- 데이터 전달 경로(`{previous}`, `_workspace/` 파일, 반환값)에 빈 구간(dead link)이 없는지 확인
- 모든 에이전트의 입력이 이전 단계의 출력과 매칭되는지 확인
- 에러 시나리오별 폴백 경로가 실행 가능한지 확인

#### 6-6. 테스트 시나리오 작성

- 오케스트레이션 프롬프트(또는 오케스트레이터 스킬)에 `## 테스트 시나리오` 섹션 추가
- 정상 흐름 1개 + 에러 흐름 1개 이상 기술

### Phase 7: 하네스 진화

하네스는 한 번 만들고 끝나는 정적 산출물이 아니다. 사용자 피드백에 따라 계속 진화하는 시스템이다.

#### 7-1. 실행 후 피드백 수집

매 하네스 실행 완료 후, 사용자에게 피드백을 요청한다:
- "결과에서 개선할 부분이 있나요?"
- "에이전트 구성이나 워크플로우에 바꾸고 싶은 점이 있나요?"

피드백이 없으면 넘어간다. 강요하지 않되, 반드시 기회를 제공한다.

#### 7-2. 피드백 반영 경로

| 피드백 유형 | 수정 대상 | 예시 |
|-----------|----------|------|
| 결과물 품질 | 해당 에이전트의 스킬 | "분석이 너무 피상적" → 스킬에 깊이 기준 추가 |
| 에이전트 역할 | 에이전트 정의 `.pi/agents/*.md` | "보안 검토도 필요" → 새 에이전트 추가 |
| 워크플로우 순서 | 오케스트레이션 프롬프트 | "검증을 먼저 해야" → chain 순서 변경 |
| 팀 구성 | 프롬프트 + 에이전트 | "이 둘은 합쳐도 될 듯" → 에이전트 병합 |
| 트리거 누락 | 스킬/프롬프트 description | "이 표현으로 하면 작동 안 함" → description 확장 |
| 모델 비용 | 에이전트 `model` frontmatter | "정찰은 더 싸게" → haiku로 다운티어 |

#### 7-3. 변경 이력

모든 변경은 AGENTS.md의 **변경 이력** 테이블에 기록한다 (Phase 5-4 템플릿과 동일):

```markdown
**변경 이력:**
| 날짜 | 변경 내용 | 대상 | 사유 |
|------|----------|------|------|
| 2026-04-05 | 초기 구성 | 전체 | - |
| 2026-04-07 | QA 에이전트 추가 | agents/qa-inspector.md | 산출물 품질 검증 부족 피드백 |
| 2026-04-10 | 정찰 모델 다운티어 | agents/scout.md | 비용 절감 |
```

이 이력을 통해 하네스가 어떤 방향으로 진화했는지 추적하고, 퇴행(regression)을 방지한다.

#### 7-4. 진화 트리거

사용자가 명시적으로 "하네스 수정해줘"라고 할 때만이 아니라, 다음 상황에서도 진화를 제안한다:
- 같은 유형의 피드백이 2회 이상 반복될 때
- 에이전트가 반복적으로 실패하는 패턴이 발견될 때
- 사용자가 오케스트레이션 프롬프트를 우회하여 수동으로 작업하는 것이 관찰될 때

#### 7-5. 운영/유지보수 워크플로우

기존 하네스의 점검·수정·동기화를 체계적으로 수행한다. Phase 0에서 "운영/유지보수" 분기로 진입했을 때 이 워크플로우를 따른다.

**Step 1: 현황 감사**
- `.pi/agents/` 파일 목록과 오케스트레이션 프롬프트가 참조하는 에이전트 구성 비교 → 불일치 목록
- `.pi/skills/` 디렉토리 목록과 에이전트가 참조하는 스킬 비교 → 불일치 목록
- 감사 결과를 사용자에게 보고한다

**Step 2: 점진적 추가/수정**
- 에이전트/스킬/프롬프트 추가·수정·삭제를 한 번에 하나씩 수행하고, 각 변경 후 즉시 Step 3을 실행한다

**Step 3: AGENTS.md 변경 이력 갱신**
- 변경 이력 테이블에 날짜, 변경 내용, 대상, 사유를 기록한다

**Step 4: 변경 검증**
- 수정된 에이전트/스킬/프롬프트의 구조 검증 (Phase 6-1 기준)
- 수정 범위가 트리거에 영향을 주면 트리거 검증 (Phase 6-4 기준)
- 대규모 변경(아키텍처 변경, 에이전트 3개 이상 추가/삭제) 시 Phase 6-3(실행 테스트), 6-5(드라이런)까지 수행
- AGENTS.md와 실제 파일의 일치 여부 최종 확인
