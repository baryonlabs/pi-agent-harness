# Agent Design Patterns (pi)

이 하네스는 **pi 코딩 에이전트**를 타깃으로 한다. pi는 빌트인 서브에이전트·실시간 팀 메시징이 없으므로, 협업은 번들된 `subagent` 확장이 등록하는 `subagent` 도구로 **별도 pi 프로세스를 스폰**하여 구현한다. 각 위임은 격리된 컨텍스트 윈도우를 가진다.

## Claude Code → pi 매핑 요약

| Claude Code 개념 | pi 대응 |
|------------------|---------|
| `.claude/agents/{name}.md` | `.pi/agents/{name}.md` (frontmatter: name/description/tools/model, 본문=시스템 프롬프트) |
| `.claude/skills/` | `.pi/skills/` (Agent Skills 표준) |
| `Agent` 도구 / `subagent_type` | `subagent` 도구 + 에이전트 이름 (`{ agent, task }`) |
| 빌트인 타입 `general-purpose`/`Explore`/`Plan` | 없음 → `tools` 화이트리스트로 권한 제어 |
| `TeamCreate` + 실시간 `SendMessage` | 없음 → 메인이 중간 산출물을 읽고 다음 위임 조정 |
| `TaskCreate`/`TaskUpdate` 공유 작업 목록 | 없음 → 메인 에이전트가 위임 흐름 관리 (`_workspace/` 파일) |
| `run_in_background` 병렬 | `subagent` 도구의 `parallel` 모드 |
| 순차 파이프라인 | `subagent` 도구의 `chain` 모드 (`{previous}`) |
| `model: "opus"` 고정 | 에이전트별 `model` 티어링 (haiku/sonnet/opus) |
| `CLAUDE.md` | `AGENTS.md` (pi 기본; CLAUDE.md도 로드됨) |

## 위임 모드: single / parallel / chain

pi의 협업 단위는 "팀"이 아니라 **위임(delegation)**이다. `subagent` 도구는 세 모드를 지원한다.

### single — 단일 위임

메인 에이전트가 하나의 에이전트에게 하나의 작업을 격리 컨텍스트로 위임한다.

```
[메인] → subagent({ agent: "scout", task: "...", agentScope: "both" }) → 결과 반환
```

**적합:** 단발성 격리 작업, 전문가 풀에서 특정 전문가 호출.

### parallel — 병렬 위임 (팬아웃/팬인)

여러 에이전트가 독립 작업을 동시에 수행한다. 최대 8개 task, 동시 4개 실행. 각 task의 최종 출력이 메인에게 반환되며 task당 50KB로 캡된다(큰 산출물은 파일로 저장 지시).

```
              ┌→ [전문가A] ─┐
[메인] parallel ├→ [전문가B] ─┼→ 메인이 결과 수집·통합
              └→ [전문가C] ─┘
```

**적합:** 동일 입력에 대한 서로 다른 관점/영역의 독립 분석.

### chain — 연쇄 위임 (파이프라인)

순차 의존 작업. 이전 단계의 출력이 `{previous}` 플레이스홀더로 다음 task에 주입된다. 첫 실패 단계에서 멈춘다.

```
[메인] chain: [scout] → [planner]{previous} → [worker]{previous} → 결과
```

**적합:** 각 단계가 이전 산출물에 강하게 의존하는 흐름.

### 모드 선택 의사결정 트리

```
작업이 순차 의존인가?
├── Yes → chain 모드 ({previous}로 단계 전달)
└── No → 독립 작업이 여럿인가?
          ├── Yes → parallel 모드 (≤8개, 메인이 통합)
          └── No (단일) → single 모드
복합이면 → 오케스트레이션 프롬프트에서 모드 조합
          (예: parallel 수집 → 메인 통합 → chain 검증)
```

> **pi의 한계와 보완:** Claude의 에이전트 팀은 팀원이 `SendMessage`로 실시간 토론·교차검증했다. pi에는 이 채널이 없다. 대신 **메인 에이전트가 통합·조정자(coordinator) 역할**을 맡아, 각 위임 결과를 읽고 다음 위임의 task를 동적으로 구성한다. "팀원 간 발견 공유"는 "메인이 A의 발견을 B의 task에 명시적으로 끼워 넣기"로 구현한다.

### 구조화된 핸드오프 스키마

pi엔 실시간 메시징이 없으므로 파일 핸드오프가 조율의 전부다. 자유서술 파일은 다음 에이전트가 무엇을 읽어야 할지 모호해진다. 따라서 각 에이전트의 **최종 반환**과 `_workspace/` 산출물 상단에 고정 스키마를 쓴다 (zob-harness의 parent-visible 메시지 포맷을 파일 기반으로 채용):

```markdown
## HANDOFF
- CONTEXT: 무엇을 했는가 (1–2줄)
- OUTPUT: 산출물 경로 + 핵심 결과 요약
- EVIDENCE: 파일:라인, 명령 결과, 근거 (검증 가능하게)
- OPEN: 미해결·불확실 항목
- NEXT: 다음 에이전트가 해야 할 것 (있으면)
```

이 스키마는 메인(조정자)이 "무엇이 끝났고, 증거는 무엇이며, 다음은 무엇인지"를 빠르게 파악하게 하여, 실시간 통신 부재를 파일로 메운다. 에이전트 정의 본문의 "입력/출력 프로토콜"에 이 스키마 사용을 명시한다. parallel 반환은 50KB로 캡되므로 HANDOFF는 요약, 상세는 파일에 둔다.

## 아키텍처 유형 (pi 위임 모드 매핑)

### 1. 파이프라인 (Pipeline) → chain
순차적 작업 흐름. 이전 에이전트의 출력이 다음 에이전트의 입력.

```
[분석] → [설계] → [구현] → [검증]
```

**적합한 경우:** 각 단계가 이전 단계의 산출물에 강하게 의존
**예시:** 소설 집필 — 세계관 → 캐릭터 → 플롯 → 집필 → 편집
**pi 구현:** `chain` 한 번으로 전체를 엮고 `{previous}`로 전달. 단계가 많고 분기가 있으면 메인이 chain을 나눠 호출하며 중간 산출물을 `_workspace/`에 보존.
**주의:** chain은 첫 실패 단계에서 멈춘다. 메인이 부분 결과로 복구할지 판단.

### 2. 팬아웃/팬인 (Fan-out/Fan-in) → parallel
병렬 처리 후 결과 통합. 독립적 작업을 동시 수행.

```
         ┌→ [전문가A] ─┐
[분배] → ├→ [전문가B] ─┼→ [메인: 통합]
         └→ [전문가C] ─┘
```

**적합한 경우:** 동일 입력에 대해 서로 다른 관점/영역의 분석이 필요
**예시:** 종합 리서치 — 공식/미디어/커뮤니티/배경 동시 조사 → 통합 보고
**pi 구현:** `parallel`로 동시 위임. **통합은 메인 에이전트의 책임**이다(pi엔 팀 합의가 없음). 각 위임은 결과를 `_workspace/02_{agent}_raw.md`에 저장하고, 메인이 전부 읽어 종합한다. 통합 품질이 전체 품질을 결정하므로, 통합 단계를 별도 reviewer 에이전트에게 chain으로 한 번 더 맡기는 것을 고려.

### 3. 전문가 풀 (Expert Pool) → single
상황에 따라 적절한 전문가를 선택 호출.

```
[메인/라우터] → subagent({ agent: 전문가A | B | C })
```

**적합한 경우:** 입력 유형에 따라 다른 처리가 필요
**예시:** 코드 리뷰 — 보안/성능/아키텍처 전문가 중 해당 영역만 호출
**pi 구현:** 메인이 입력을 분류한 뒤 `single`로 해당 전문가만 호출. 여러 전문가가 동시에 필요하면 `parallel`.

### 4. 생성-검증 (Producer-Reviewer) → chain
생성 에이전트와 검증 에이전트가 쌍으로 동작.

```
[worker] → [reviewer] → (문제시) → [worker] 재작업
```

**적합한 경우:** 산출물의 품질 보장이 중요하고 객관적 검증 기준이 존재
**예시:** pi 공식 `implement-and-review` 프롬프트 — worker → reviewer → worker
**pi 구현:** `chain: [worker, reviewer, worker]`. reviewer의 지적이 `{previous}`로 마지막 worker에 전달된다. 무한 루프 방지를 위해 재작업은 1~2회로 제한(chain 길이로 고정).

### 5. 감독자 (Supervisor) → 메인의 동적 위임 루프
메인 에이전트가 진행 상황을 보며 하위 에이전트에 동적으로 작업을 분배.

```
         ┌→ [워커A]
[메인] ───┼→ [워커B]    ← 메인이 결과를 보고 다음 배치를 동적 결정
         └→ [워커C]
```

**적합한 경우:** 작업량이 가변적이거나 런타임에 작업 분배를 결정해야 할 때
**예시:** 대규모 코드 마이그레이션 — 메인이 파일 목록을 분석하고 워커들에게 배치 할당
**pi 구현:** 공유 작업 목록(TaskCreate)이 없으므로 **메인이 직접 루프**를 돈다: 파일 배치를 만들고 `parallel`로 위임 → 결과 수집 → 남은 작업이 있으면 다음 배치를 다시 `parallel` 위임. 진행 상태는 메인의 컨텍스트(또는 `_workspace/progress.md`)로 추적.

### 6. 계층적 위임 (Hierarchical Delegation) → 2단계
상위 에이전트가 하위 에이전트에 위임. 복잡한 문제를 단계적으로 분해.

```
[메인] → subagent(팀장A) → 팀장A가 다시 subagent(실무자A1, A2)
```

**적합한 경우:** 문제가 자연스럽게 계층적으로 분해되는 구조
**pi 구현:** 위임된 에이전트도 `subagent` 도구를 쓸 수 있다면 2단계 위임이 가능하다(번들 확장이 위임된 프로세스에도 로드되는 경우). 다만 깊이가 깊어지면 지연·컨텍스트 손실·비용이 커지므로 **2단계 이내**를 권장한다. 대안: 메인이 평탄화하여 직접 모든 위임을 관리.

## 복합 패턴

실전에서는 단일 패턴보다 복합 패턴이 흔하다:

| 복합 패턴 | pi 구성 | 예시 |
|----------|---------|------|
| **팬아웃 + 생성-검증** | parallel 생성 → 각각 chain(생성→리뷰) | 다국어 번역 — 4개 언어 병렬 → 각각 네이티브 리뷰어 검수 |
| **파이프라인 + 팬아웃** | chain 중 한 단계를 parallel로 | 분석(single) → 구현(parallel) → 통합 테스트(single) |
| **감독자 + 전문가 풀** | 메인 루프가 분류 후 single 동적 호출 | 고객 문의 처리 — 메인이 분류 후 적합 전문가 위임 |

복합 패턴은 단일 오케스트레이션 프롬프트가 여러 `subagent` 호출(모드 혼합)을 순서대로 지시하는 형태로 구현한다. 각 단계 사이의 데이터는 `{previous}`(chain 내부) 또는 `_workspace/` 파일(단계 간)로 전달한다.

## 에이전트 정의 구조 (pi)

모든 에이전트는 `프로젝트/.pi/agents/{name}.md` 파일로 정의한다. 본문이 곧 위임된 pi 프로세스의 **시스템 프롬프트**가 된다.

```markdown
---
name: agent-name
description: 이 에이전트가 무엇을 하는지 (subagent 도구가 위임 대상 선택 시 참조)
tools: read, grep, find, ls
model: claude-sonnet-4-5
---

당신은 [도메인]의 [역할] 전문가입니다.

## 핵심 역할
1. 역할1
2. 역할2

## 작업 원칙
- 원칙1
- 원칙2

## 입력/출력 프로토콜
- 입력: [task로 무엇을 받는지, 이전 산출물 경로]
- 출력: [어디에 무엇을 쓰는지 — 예: _workspace/01_agent_artifact.md]
- 형식: [파일 포맷, 구조]
- 최종 출력: [메인에게 반환할 요약. parallel은 50KB 캡이므로 핵심만]

## 사용 스킬 (선택)
- `/skill:{name}`을 로드하여 {작업}을 수행하거나, 본문에 절차를 인라인

## 재호출 지침 (후속 작업)
- 이전 산출물 파일이 존재하면 읽고 개선점을 반영
- 피드백이 주어지면 해당 부분만 수정

## 에러 핸들링
- [실패/모호함 시 행동]
```

### frontmatter 필드

| 필드 | 필수 | 설명 |
|------|------|------|
| `name` | 예 | 위임 시 호출 이름. 파일명과 달라도 됨 |
| `description` | 예 | 위임 대상 선택 단서 |
| `tools` | 아니오 | 공백/쉼표 구분 화이트리스트. 생략 시 기본 도구(read/write/edit/bash) 전체 |
| `model` | 아니오 | 작업 난도에 맞춰 티어링 (아래) |

### tools 화이트리스트로 권한 제어

pi엔 `Explore`/`Plan` 같은 읽기 전용 빌트인 타입이 없다. 대신 `tools`로 동일 효과를 낸다:

| 역할 | 권장 tools | 효과 |
|------|-----------|------|
| 읽기 전용 분석/리뷰 | `read, grep, find, ls` | 파일 수정 불가 (Explore 대체) |
| 읽기 전용 + git 조회 | `read, grep, find, ls, bash` | bash는 본문에서 read-only 명령(`git diff` 등)만 지시 (reviewer/QA) |
| 계획 수립 | `read, grep, find, ls` | 코드 변경 없이 설계 (Plan 대체) |
| 구현(파일 수정) | 생략 또는 `read, write, edit, bash, grep, find, ls` | 전체 도구 |

> **주의:** 도구 권한은 완벽히 강제되지 않을 수 있다. bash를 부여한 read-only 에이전트에는 본문에서 "bash는 read-only 명령만 사용"을 명시한다.

### model 티어링

모든 에이전트를 opus로 고정하던 Claude 하네스와 달리, pi는 작업 난도에 맞춰 모델을 고른다:

| 작업 | 권장 model | 이유 |
|------|-----------|------|
| 정찰/추출/단순 수집 | `claude-haiku-4-5` | 빠르고 저렴, 충분한 품질 |
| 일반 구현/리뷰/분석 | `claude-sonnet-4-5` | 균형 |
| 고난도 설계/통합/추론 | `claude-opus-4-7` | 최고 품질, 결과 좌우 단계 |

추론 품질이 최종 결과를 좌우하는 에이전트(통합자, 아키텍트, 핵심 구현자)는 opus로 올리고, 기계적 수집은 haiku로 내려 비용을 절감한다.

## 에이전트 분리 기준

| 기준 | 분리 | 통합 |
|------|------|------|
| 전문성 | 영역이 다르면 분리 | 영역이 겹치면 통합 |
| 병렬성 | 독립 실행 가능하면 분리 (parallel 활용) | 순차 종속이면 통합 고려 |
| 컨텍스트 | 컨텍스트 부담이 크면 분리 (격리 위임의 이점) | 가볍고 빠르면 통합 |
| 재사용성 | 다른 워크플로우에서도 쓰면 분리 | 이 워크플로우에서만 쓰면 통합 고려 |

## 에이전트 재사용 설계

신규 에이전트 생성 전, 기존 에이전트와의 중복을 확인한다. 하네스를 반복 구축하다 보면 역할이 겹치는 에이전트가 다른 이름으로 누적되기 쉽다.

| 상황 | 조치 |
|------|------|
| 기존 에이전트가 신규 역할을 완전히 포함 | 신규 생성 금지 — 기존 에이전트 재사용 |
| 기존 에이전트가 부분 포함이고 일반화 가능 | 기존 에이전트를 일반화하여 확장 |
| 도메인 특화가 의도된 부분 포함 | 신규 생성 진행 — 별개 에이전트로 유지 |
| 역할 범위가 완전히 다름 | 신규 생성 진행 |

**원칙:** 하나의 에이전트가 하나의 역할에 집중할수록 재사용성이 높고 중복이 줄어든다. 역할이 두 가지 이상이면 분리할 수 있는지 먼저 검토한다.

**기존 에이전트 일반화 시:** 해당 에이전트를 참조하는 오케스트레이션 프롬프트의 동작이 변경될 수 있다. 확장 전 의존성을 확인하고, 일반화 후 드라이런으로 기존 동작 유지를 확인한다.

## 스킬 vs 에이전트 구분

| 구분 | 스킬 (Skill) | 에이전트 (Agent) |
|------|-------------|-----------------|
| 정의 | 절차적 지식 + 도구 번들 | 전문가 페르소나 + 행동 원칙 (위임 프로세스의 시스템 프롬프트) |
| 위치 | `.pi/skills/` | `.pi/agents/` |
| 트리거 | 시스템 프롬프트 노출 + 키워드 매칭, 또는 `/skill:name` | `subagent` 도구로 명시적 위임 |
| 크기 | 작은~큰 (워크플로우) | 작은 (역할 정의) |
| 용도 | "어떻게 하는가" | "누가 하는가" |

스킬은 에이전트가 작업을 수행할 때 참조하는 **절차적 가이드**. 에이전트는 스킬을 활용하는 **전문가 역할 정의**.

## 스킬 ↔ 에이전트 연결 방식

위임된 서브에이전트는 격리된 pi 프로세스이므로, 사용할 스킬을 명확히 지정해야 로드된다:

| 방식 | 구현 | 적합한 경우 |
|------|------|-----------|
| **/skill 로드** | 위임 task에 `/skill:name`을 포함하거나, 에이전트 본문에서 "작업 전 /skill:name을 read로 로드" 지시 | 스킬이 독립 워크플로우이고 재사용성이 높은 경우 |
| **프롬프트 내 인라인** | 에이전트 정의 본문에 절차를 직접 포함 | 절차가 짧고(50줄 이하) 이 에이전트 전용인 경우 |
| **레퍼런스 로드** | 본문에서 "필요 시 references/{x}.md를 read로 로드" 지시 | 스킬 내용이 크고 조건부로만 필요한 경우 |

권장: 재사용성이 높으면 `/skill` 로드, 전용이면 인라인, 대용량이면 레퍼런스 로드.
